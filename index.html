1. Executive Summary

JobFlow คือระบบบริหารงานผลิตตามออเดอร์ สำหรับร้านป้าย ร้านสติ๊กเกอร์ โรงพิมพ์ ร้านสกรีน ร้านเลเซอร์คัต ร้าน CNC และธุรกิจรับผลิตตามสั่ง

เป้าหมายหลักคือทำให้ทุกงานมี “ศูนย์กลางข้อมูลเดียว” ตั้งแต่รับออเดอร์จนปิดงาน

จากเดิมที่ข้อมูลกระจายอยู่ใน

  • Google Sheet สำหรับคิวงาน
  • Dropbox สำหรับไฟล์
  • LINE สำหรับส่งแบบและอนุมัติ

JobFlow จะรวมข้อมูลสำคัญไว้ในระบบเดียว ได้แก่

  • ข้อมูลลูกค้า
  • ข้อมูลงาน
  • สถานะงาน
  • ผู้รับผิดชอบ
  • ลิงก์ไฟล์ Dropbox
  • หลักฐานอนุมัติจากลูกค้า
  • Timeline การทำงาน
  • Dashboard ภาพรวมธุรกิจ

แนวทาง CTO คือสร้างระบบให้เป็น Multi-Tenant SaaS Ready ตั้งแต่วันแรก แม้ Version 1 จะใช้กับร้านเดียวก่อน เพื่อให้อนาคตรองรับหลายร้าน หลายธุรกิจ หลายแพ็กเกจ โดยไม่ต้องรื้อระบบใหม่

2. Product Vision

Vision

JobFlow คือระบบกลางที่ทำให้เจ้าของร้านรู้ทันทีว่า

  • งานอยู่ตรงไหน
  • ใครรับผิดชอบ
  • ไฟล์อยู่ที่ไหน
  • ลูกค้าอนุมัติหรือยัง
  • งานไหนกำลังจะช้า
  • งานไหนเสร็จแล้ว
  • งานไหนตกหล่น

Core Value Proposition

สำหรับเจ้าของร้าน

เห็นภาพรวมธุรกิจทันที ไม่ต้องถามพนักงานทีละคน

สำหรับผู้จัดการ

ควบคุมคิวงาน ติดตามสถานะ และแก้ปัญหางานล่าช้าได้เร็ว

สำหรับฝ่ายขาย

เปิดงานได้เป็นระบบ ค้นหาลูกค้าและงานย้อนหลังได้ง่าย

สำหรับฝ่ายออกแบบ

รู้ว่างานไหนต้องออกแบบ งานไหนรออนุมัติ งานไหนแก้แบบ

สำหรับฝ่ายผลิต

เห็นงานที่พร้อมผลิต พร้อมพิมพ์ พร้อมตัด และลำดับความสำคัญ

3. Problems to Solve

ปัญหาหลัก

ปัญหา ผลกระทบ JobFlow แก้อย่างไร
หาไฟล์ไม่เจอ งานล่าช้า พิมพ์ผิดไฟล์ ผูก Dropbox URL กับเลขงาน
หาเลขงานยาก สื่อสารผิดพลาด มีเลขงานมาตรฐาน C00001
ไม่รู้สถานะงาน ต้องถามกันตลอด Status Pipeline ชัดเจน
งานตกหล่น เสียลูกค้า Dashboard + Overdue Alert
ลูกค้าตามงานบ่อย เสียเวลาตอบ ค้นหางานและสถานะได้ทันที
ความรู้ติดอยู่กับคน เปลี่ยนพนักงานแล้วสะดุด Timeline บันทึกทุกเหตุการณ์
ใช้หลายระบบแยกกัน ข้อมูลกระจัดกระจาย รวมข้อมูลสำคัญใน Job เดียว

4. Target Users

4.1 Owner / เจ้าของร้าน

ต้องการเห็นภาพรวมทั้งหมด เช่น จำนวนงาน งานล่าช้า งานรออนุมัติ งานกำลังผลิต

4.2 Manager / ผู้จัดการ

ต้องการจัดคิว มอบหมายงาน เปลี่ยนสถานะ และติดตามงานทุกฝ่าย

4.3 Sales / ฝ่ายขาย

ต้องการเปิดงานใหม่ ค้นหาลูกค้า ดูประวัติ และบันทึกรายละเอียดงาน

4.4 Designer / ฝ่ายออกแบบ

ต้องการเห็นงานที่ต้องออกแบบ อัปเดตไฟล์แบบ และส่งสถานะรออนุมัติ

4.5 Production / ฝ่ายผลิต

ต้องการเห็นงานที่อนุมัติแล้ว พร้อมพิมพ์ พร้อมตัด พร้อมส่งมอบ

4.6 Future User: Customer

ใน Version 2 หรือ 3 สามารถให้ลูกค้าดูสถานะงานผ่านลิงก์ หรือดู Catalog ผลงานของร้านได้

8. Job Status Design

8.1 สถานะงาน

Status Code ชื่อสถานะ ความหมาย
RECEIVED รับงาน เปิดงานใหม่แล้ว
DESIGNING ออกแบบ ฝ่ายออกแบบกำลังทำ
WAITING_APPROVAL รออนุมัติ ส่งแบบให้ลูกค้ารออนุมัติ
APPROVED อนุมัติแล้ว ลูกค้าอนุมัติแบบแล้ว
PRINTING กำลังพิมพ์ เริ่มผลิตงานพิมพ์
CUTTING กำลังตัด อยู่ขั้นตอนตัด
READY_DELIVERY พร้อมส่งมอบ ผลิตเสร็จ รอส่ง
DELIVERED ส่งมอบแล้ว ส่งงานให้ลูกค้าแล้ว
CLOSED ปิดงาน งานจบสมบูรณ์

14. Database Schema Design

14.1 Design Principle

แม้ MVP จะใช้ร้านเดียว แต่ Database ต้องมี tenant_id ตั้งแต่แรก

เพราะอนาคต JobFlow จะเป็น SaaS ที่มีหลายร้าน หลายธุรกิจ

Core Rule:

  • ทุก table หลักต้องมี tenant_id
  • ทุกข้อมูลต้องแยกตาม tenant
  • ห้าม query ข้าม tenant
  • ทุก API ต้องตรวจ tenant จาก JWT

16. PostgreSQL Schema

CREATE TABLE tenants (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  business_type VARCHAR(100),
  phone VARCHAR(50),
  line_id VARCHAR(100),
  is_active BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMP DEFAULT now(),
  updated_at TIMESTAMP DEFAULT now()
);

CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  full_name VARCHAR(255) NOT NULL,
  email VARCHAR(255) NOT NULL,
  password_hash TEXT NOT NULL,
  phone VARCHAR(50),
  is_active BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMP DEFAULT now(),
  updated_at TIMESTAMP DEFAULT now(),
  UNIQUE (tenant_id, email)
);

CREATE TABLE roles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code VARCHAR(50) NOT NULL UNIQUE,
  name VARCHAR(100) NOT NULL
);

CREATE TABLE user_roles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  user_id UUID NOT NULL REFERENCES users(id),
  role_id UUID NOT NULL REFERENCES roles(id),
  created_at TIMESTAMP DEFAULT now(),
  UNIQUE (tenant_id, user_id, role_id)
);

CREATE TABLE customers (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  customer_code VARCHAR(50) NOT NULL,
  name VARCHAR(255) NOT NULL,
  phone VARCHAR(50),
  line_id VARCHAR(100),
  note TEXT,
  created_by UUID REFERENCES users(id),
  created_at TIMESTAMP DEFAULT now(),
  updated_at TIMESTAMP DEFAULT now(),
  deleted_at TIMESTAMP,
  UNIQUE (tenant_id, customer_code)
);

CREATE TABLE job_sequences (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  prefix VARCHAR(10) NOT NULL DEFAULT 'C',
  current_number INT NOT NULL DEFAULT 0,
  created_at TIMESTAMP DEFAULT now(),
  updated_at TIMESTAMP DEFAULT now(),
  UNIQUE (tenant_id, prefix)
);

CREATE TYPE job_status AS ENUM (
  'RECEIVED',
  'DESIGNING',
  'WAITING_APPROVAL',
  'APPROVED',
  'PRINTING',
  'CUTTING',
  'READY_DELIVERY',
  'DELIVERED',
  'CLOSED',
  'CANCELLED',
  'ON_HOLD',
  'REVISION'
);

CREATE TYPE job_priority AS ENUM (
  'LOW',
  'NORMAL',
  'HIGH',
  'URGENT'
);

CREATE TABLE jobs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  job_number VARCHAR(50) NOT NULL,
  title VARCHAR(255) NOT NULL,
  description TEXT,
  customer_id UUID NOT NULL REFERENCES customers(id),
  received_date DATE NOT NULL DEFAULT CURRENT_DATE,
  due_date DATE,
  priority job_priority DEFAULT 'NORMAL',
  status job_status DEFAULT 'RECEIVED',
  assigned_to UUID REFERENCES users(id),
  created_by UUID REFERENCES users(id),
  closed_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT now(),
  updated_at TIMESTAMP DEFAULT now(),
  deleted_at TIMESTAMP,
  UNIQUE (tenant_id, job_number)
);

CREATE TYPE job_file_type AS ENUM (
  'DESIGN_FILE',
  'PRINT_FILE',
  'CUT_FILE',
  'APPROVAL_IMAGE',
  'OTHER'
);

CREATE TABLE job_files (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  job_id UUID NOT NULL REFERENCES jobs(id),
  file_type job_file_type NOT NULL,
  variant_name VARCHAR(100),
  file_name VARCHAR(255),
  dropbox_folder_url TEXT,
  dropbox_file_url TEXT,
  note TEXT,
  uploaded_by UUID REFERENCES users(id),
  created_at TIMESTAMP DEFAULT now(),
  updated_at TIMESTAMP DEFAULT now()
);

CREATE TABLE job_approvals (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  job_id UUID NOT NULL REFERENCES jobs(id),
  approval_image_url TEXT,
  line_screenshot_url TEXT,
  approved_at TIMESTAMP,
  approved_by_customer_name VARCHAR(255),
  note TEXT,
  created_by UUID REFERENCES users(id),
  created_at TIMESTAMP DEFAULT now(),
  updated_at TIMESTAMP DEFAULT now()
);

CREATE TYPE timeline_event_type AS ENUM (
  'JOB_CREATED',
  'JOB_UPDATED',
  'STATUS_CHANGED',
  'FILE_ADDED',
  'APPROVAL_ADDED',
  'ASSIGNEE_CHANGED',
  'PRODUCTION_STARTED',
  'DELIVERED',
  'JOB_CLOSED',
  'NOTE_ADDED'
);

CREATE TABLE job_timelines (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  job_id UUID NOT NULL REFERENCES jobs(id),
  event_type timeline_event_type NOT NULL,
  title VARCHAR(255) NOT NULL,
  description TEXT,
  metadata JSONB,
  created_by UUID REFERENCES users(id),
  created_at TIMESTAMP DEFAULT now()
);

CREATE TABLE job_status_histories (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  job_id UUID NOT NULL REFERENCES jobs(id),
  from_status job_status,
  to_status job_status NOT NULL,
  changed_by UUID REFERENCES users(id),
  changed_at TIMESTAMP DEFAULT now(),
  note TEXT
);

28. Key Technical Decisions

28.1 ใช้ Modular Monolith ก่อน

ไม่ควรเริ่มด้วย Microservices เพราะ MVP จะช้าและแพงเกินไป

28.2 ใส่ tenant_id ทุก table ตั้งแต่วันแรก

28.3 ไม่เก็บไฟล์จริงใน V1

28.4 Timeline ต้องเป็นหัวใจของระบบ

ทุก action สำคัญต้องถูกบันทึก เพื่อให้ตามย้อนหลังได้

28.5 Search ต้องเร็ว

JobFlow SaaS Web Application

Product Requirement & System Design Document

1. Executive Summary

JobFlow is a job management system for made-to-order production businesses such as sign shops, sticker shops, print shops, screen-printing shops, laser-cutting shops, CNC shops, and other custom-production businesses.

The main goal is to give every job a single source of truth from order intake through job closing.

Today, information is spread across:

  • Google Sheets for the job queue
  • Dropbox for files
  • LINE for artwork delivery and approval

JobFlow consolidates the key information in one system, including:

  • Customer information
  • Job details
  • Job status
  • Responsible person
  • Dropbox file links
  • Customer approval evidence
  • Work timeline
  • Business overview dashboard

The CTO direction is to build the system as Multi-Tenant SaaS Ready from day one, even if Version 1 is used by only one shop first. This makes it possible to support multiple shops, multiple business types, and multiple packages in the future without rebuilding the system.

CREATE TABLE tenants (

  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  full_name VARCHAR(255) NOT NULL,

  phone VARCHAR(50),

  created_at TIMESTAMP DEFAULT now(),

30. Final CTO Recommendation

For JobFlow, the MVP should start as follows:

  1. Build a system that works for one shop first
  2. Design the database as multi-tenant from day one
  3. Use Next.js + NestJS + PostgreSQL + Docker
  4. Do not store files directly in V1; use Dropbox URLs first
  5. Make Job Detail the central place for all job information
  6. Make Timeline record every important event
  7. Make Dashboard the owner’s first screen
  8. Make Search as fast as possible because it addresses a major pain point
  9. Build permission control from the beginning for multiple departments
  10. Plan Version 2 as Customer Portal + Catalog + Notification
  11. Plan Version 3 as full SaaS + Billing + Multi-Branch + LINE OA + Cost Tracking

16. Auth & Security

16.1 Authentication

  • ใช้ Supabase Auth — Email/Password สำหรับแอดมิน
  • ไม่มีระบบสมัครสมาชิกสำหรับลูกค้า
  • Session จัดการผ่าน Supabase Client + @supabase/ssr
  • Middleware ของ Next.js ตรวจสอบ session ก่อนเข้า /admin/*

16.2 Authorization

  • Public Route: /, /portfolio, /catalog, /services, /contact, /api/* (GET)
  • Protected Route: /admin/*, /api/admin/* — ต้อง Login

16.3 Supabase RLS (Row Level Security)

  • Public Read: is_published = true สำหรับ portfolio_items และ catalog_items
  • Admin Full Access: ผู้ใช้ที่ Authenticated สามารถ CRUD ได้ทุก row
  • site_settings: Public Read (ทุกคน), Write เฉพาะ Authenticated

16.4 File Upload Security

  • ตรวจสอบ file type: เฉพาะ image/jpeg, image/png, image/webp
  • จำกัดขนาดไฟล์: ไม่เกิน 5 MB ต่อไฟล์
  • เก็บใน Private Bucket — เข้าถึงผ่าน Signed URL หรือ Public URL ที่ตั้งไว้

1. Executive Summary

Print Shop Website is the main website for a custom print shop, designed to let prospective customers view work quality, service options, and pricing before reaching out — without having to ask basic questions first.

The core problem: customers don't know what the shop produces, how much it costs, or what quality to expect. They must contact the shop each time before placing an order, consuming significant staff time answering repetitive questions.

The system consists of two main parts:

  • Public Website — Customer-facing pages showcasing portfolio work, product catalog, service specs, and starting prices.
  • Admin CMS — A content management system letting admins add, edit, delete, and upload images without touching code.

Reduce time spent answering repetitive questions, increase credibility, and attract new customers via Google search.

2. PHP 8.3 — Laravel / Vanilla

PHP 8.3
ตัวเลือกที่คุ้นเคย
ใช้ได้ทั้ง Vanilla PHP 8.3/PDO (แบบที่ทำ Product Catalog CMS) หรือ Laravel 11/12 (แบบที่เริ่ม ERP)

✅ ข้อดี

  • Deploy ง่ายที่สุด — upload ขึ้น VPS / shared hosting ตรงๆ ได้เลย
  • ไม่ต้องใช้ Docker / Node.js / npm — dev บน XAMPP ได้ทันที
  • มีประสบการณ์จริง — ทำ Product Catalog CMS ด้วย PHP/PDO/MySQL/AdminLTE มาแล้ว
  • Server-rendered HTML ทุกครั้ง → SEO ดีโดย default (ไม่มี hydration gap)
  • RAM บน production ต่ำ (PHP-FPM ใช้ RAM ต่อ request เป็น ephemeral)
  • ทำงานดีบน HP ProBook 645 G1 — ไม่มี build step ที่หนัก
  • Laravel: มี Filament หรือ AdminLTE สำหรับ Admin UI สวยๆ
  • Laravel: Eloquent ORM, Spatie RBAC, Livewire/InertiaJS สำหรับ SPA-feel
  • PHP 8.3 มี type declaration, readonly properties, Fibers → ไม่เก่าอีกต่อไป

❌ ข้อเสีย

  • Frontend interactivity ต้องจัดการแยก (Alpine.js, htmx, หรือ Livewire)
  • ไม่มี type safety บน frontend (ต้องพึ่ง JS/TS แยก)
  • SaaS multi-tenant ทำได้แต่ต้องออกแบบดีมาก (package: stancl/tenancy)
  • Real-time WebSocket ต้องใช้ Pusher / Soketi / Reverb เพิ่ม
  • ถ้าอยากได้ rich interactive UI เทียบกับ React — ช่องว่างใหญ่มาก
  • Laravel มี learning curve (Eloquent relationship, Service Container, Facades)
  • ไม่ ideal ถ้าต้องการ type-safe API contract กับ Next.js frontend ในอนาคต
  • Vanilla PHP → boilerplate เยอะ ถ้าโปรเจกต์โตขึ้น
🎯 เหมาะกับ Print Shop Website หรือ JobFlow?
  • Print Shop Website: ✅ เหมาะมาก — content site + admin CMS เป็น sweet spot ของ PHP มาตลอด ทำเสร็จเร็ว deploy ง่าย
  • JobFlow SaaS: ⚠️ ทำได้ด้วย Laravel + Livewire/InertiaJS แต่ถ้า scope โตเป็น multi-tenant จริงๆ จะเริ่มรู้สึก friction มากกว่า TypeScript stack

3. Bun + Hono.js / Elysia.js

Bun Runtime
The lightweight TypeScript backend
Bun เป็น JS runtime ที่เร็วกว่า Node.js 3–4× Hono.js/Elysia.js เป็น micro-framework บนนั้น (คล้ายแนวทางที่ใช้ใน Credit Bank system)

✅ ข้อดี

  • เร็วมาก — startup time ต่ำมาก, throughput สูง, เหมาะกับ API-heavy workload
  • TypeScript native — ไม่ต้อง compile step เพิ่ม (Bun รัน TS ตรงๆ ได้)
  • RAM footprint ต่ำกว่า NestJS มาก (Hono/Elysia เป็น micro-framework)
  • ไม่มี Node.js overhead — build tool, test runner, bundler อยู่ใน Bun เลย
  • Hono: edge-ready, ทำงานได้ใน Cloudflare Workers / Deno / Node ด้วย
  • Elysia: type-safety แบบ end-to-end (Elysia + Eden Treaty) คล้าย tRPC
  • ดี้สำหรับ microservice หรือ standalone API backend
  • SQLite รองรับ native ใน Bun — เหมาะสำหรับ prototype ขนาดเล็ก

❌ ข้อเสีย

  • ยังต้องหา frontend framework แยก (Vite+React, Astro, SvelteKit)
  • ไม่มี SSR built-in — ถ้าต้องการ SEO ดีต้องใช้ Astro หรือ SvelteKit ควบ
  • Ecosystem ยัง mature น้อยกว่า Node.js — บาง npm package อาจ break
  • ไม่มี built-in DI, Guards, Interceptors อย่าง NestJS — ต้องประกอบ pattern เอง
  • สำหรับ Enterprise SaaS ขาด structure ที่ชัดเจน (convention over configuration)
  • Thai community support น้อยกว่า PHP/Laravel มาก
  • ถ้า Node.js package ไม่ compatible กับ Bun → ต้อง workaround
🎯 เหมาะกับ Print Shop Website หรือ JobFlow?
  • Print Shop Website: ⚠️ ทำได้ (Hono เป็น API backend + Astro เป็น frontend) แต่ overkill และต้องประกอบ 2 repo ต่างกัน ไม่คุ้มเมื่อเทียบกับ Next.js ที่รวมอยู่ในที่เดียว
  • JobFlow SaaS: ⚠️ ดีสำหรับ prototype หรือ lightweight API layer แต่ถ้า scope ขยายต้องการ RBAC, approval workflows, multi-tenant — NestJS structure มีประโยชน์กว่า Hono/Elysia มาก
  • Best use case: ✅ ใช้เป็น backend ของ internal microservice หรือ webhook handler — ไม่ใช่ main backend ของ SaaS เต็มรูปแบบ

Next.js + NestJS
Full-stack TypeScript — Enterprise ready

✅ ข้อดี

  • SSR / SSG / ISR built-in → SEO ดีที่สุดในสาม stack นี้
  • TypeScript end-to-end — type safety ตั้งแต่ database schema จนถึง UI

  • Docker Compose ทำให้ dev environment reproducible ทุกเครื่อง
  • Monorepo ทำ type sharing ระหว่าง frontend และ backend ได้
